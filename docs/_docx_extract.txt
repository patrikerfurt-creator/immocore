1. Zweck dieses Dokuments
Dieses Dokument ist der vollständige Claude Code Implementierungsprompt für das Modul „Rechnungserkennung mit Lernlogik“. Es löst die Versionen 1.0 und 1.1 ab. Erweitert werden:
drei klar getrennte Erkennungsstufen mit dimensions-spezifischem Routing
ein lernendes Regelwerk für die automatische Buchungskonto-Zuordnung
Pflichtfeld Objekt.betreuer als Routing-Ziel für objektbezogene Prüffälle
neue Rolle Frontoffice für ungeklärte Eingänge ohne erkennbares Objekt
zusätzliche 95%-Konfidenz-Schwelle als Voraussetzung für automatische Buchung
Das Modul ist additiv. Bestehende Status (in_pruefung, freigegeben, gebucht, bezahlt, abgelehnt) und der Mechanismus zahlungsfreigabe_grenzen bleiben unverändert.
Voraussetzung
Phasen 1–3 abgeschlossen. Models Rechnung, Freigabe, Konto, Person (mit Rolle Dienstleister), Objekt vorhanden. KI-OCR-Endpunkt und ZUGFeRD/XRechnung-Parser funktional. Bezug: IMMOCORE_Ausgangsspezifikation v1.1 Kap. 7 + 7.3. KI-Modell: claude-sonnet-4-6.
2. Workflow im Überblick
2.1 Drei Phasen
Erkennung und Freigabe sind sauber getrennt. Zwischen den beiden Phasen entscheidet ein Routing-Schritt anhand der erkannten Dimensionen, wer die Rechnung bearbeitet.
Phase A — Erkennung: automatisch, technisch, ohne Nutzerinteraktion.
Phase B — Routing: technisch, weist Rechnung dem richtigen Bearbeiter zu (Limit-Workflow / Objektbetreuer / Frontoffice).
Phase C — Bearbeitung: Auto-Buchung ODER manuelle Identifikation/Freigabe.

3.7 Leistungstext-Normalisierung
def normalisiere_leistungstext(text: str) -> str:
text = text.lower()
text = re.sub(r'\d{1,2}[./]\d{1,2}[./]\d{2,4}', '', text)  # Datum
text = re.sub(r'\b(q[1-4]|kw\s?\d{1,2})\b', '', text)     # Quartal/KW
text = re.sub(r'(rg|re|rechnung|beleg)[-.\s]?\d+', '', text)  # Belegnr
text = re.sub(r'[^a-z0-9\u00e4\u00f6\u00fc\u00df ]+', ' ', text)
text = re.sub(r'\s+', ' ', text).strip()
text = ' '.join(t for t in text.split() if t not in DE_STOPWORDS)
return text
def leistungstext_hash(text: str) -> str:
return hashlib.sha256(
normalisiere_leistungstext(text).encode('utf-8')
).hexdigest()

Zwei Schwellen-Schichten
1. Eindeutigkeits-Schwelle (oben) entscheidet, ob eine Dimension als 'erkannt' gilt → daraus folgt die Stufe. 2. Auto-Buchung-Schwelle (≥ 95% in allen drei Dimensionen) ist eine ZUSÄTZLICHE Hürde nur für Stufe 1 mit Betrag < Auto-Limit. Eine Stufe-1-Rechnung mit Konfidenz 92/88/100 geht NICHT auto, sondern in den Limit-Workflow.
5.3 Pseudocode — Stufenableitung
def fuehre_erkennung_aus(rechnung: Rechnung) -> Rechnung:
log = RechnungsErkennungsLog(rechnung=rechnung)
kreditor_match = match_kreditor(rechnung)
objekt_match   = match_objekt(rechnung)
konto_match, regel_treffer = MatchResult.empty(), None
if kreditor_match.eindeutig and objekt_match.eindeutig:
text_hash = leistungstext_hash(rechnung.leistungstext or '')
regel_treffer = RechnungsMatchRegel.objects.filter(
kreditor=kreditor_match.kandidat,
objekt=objekt_match.kandidat,
leistungstext_hash=text_hash,
status='aktiv',
).first()
if regel_treffer:
konto_match = MatchResult.eindeutig(
regel_treffer.buchungskonto, 1.0
)
else:
konto_match = match_konto_historie(
kreditor_match.kandidat, objekt_match.kandidat
)  # Konfidenz <= 0.7 — Vorschlag
# Stufenableitung mit Sub-Stufen 2a/2b
k_eind = kreditor_match.eindeutig
o_eind = objekt_match.eindeutig
c_eind = konto_match.eindeutig
if k_eind and o_eind and c_eind:
rechnung.status = 'erkannt'
rechnung.erkennungs_stufe = '1'
elif o_eind:
# Objekt da — unabhängig ob Kreditor/Konto fehlen
rechnung.status = 'pruefung_match'
rechnung.erkennungs_stufe = '2a'
elif k_eind:
# Nur Kreditor
rechnung.status = 'pruefung_match'
rechnung.erkennungs_stufe = '2b'
else:
rechnung.status = 'nicht_erkannt'
rechnung.erkennungs_stufe = '3'
rechnung.kreditor      = kreditor_match.kandidat if k_eind else None
rechnung.objekt        = objekt_match.kandidat   if o_eind else None
rechnung.buchungskonto = konto_match.kandidat    if c_eind else None
rechnung.match_regel   = regel_treffer
rechnung.erkennungs_konfidenz = {
'kreditor': kreditor_match.konfidenz,
'objekt':   objekt_match.konfidenz,
'konto':    konto_match.konfidenz,
}
route_rechnung(rechnung)  # Phase B
rechnung.save(); log.save()
return rechnung

6.2 route_rechnung — Pseudocode
AUTO_KONFIDENZ_SCHWELLE = 0.95
def route_rechnung(rechnung: Rechnung) -> None:
if rechnung.status == 'erkannt':
rechnung.routing_ziel = 'limit_workflow'
grenzen = rechnung.objekt.zahlungsfreigabe_grenzen
stufe = ermittle_freigabestufe(rechnung.betrag, grenzen)
if stufe['rolle'] == 'auto' and konfidenz_min(rechnung) >= AUTO_KONFIDENZ_SCHWELLE:
buche_rechnung(rechnung)  # Status -> 'gebucht'
return
# Auto-Limit gilt, aber Konfidenz unzureichend → in_pruefung
rechnung.status = 'in_pruefung'
rechnung.zugewiesen_an = ermittle_freigabeperson(
rechnung.objekt,
stufe if stufe['rolle'] != 'auto' else stufe_naechste_manuelle(grenzen),
)
starte_eskalations_task(rechnung, stufe['frist_tage'])
return
if rechnung.status in ('pruefung_match', 'nicht_erkannt'):
if rechnung.erkennungs_stufe == '2a':
rechnung.routing_ziel = 'objektbetreuer'
betreuer = rechnung.objekt.betreuer
if betreuer.abwesend and rechnung.objekt.betreuer_vertretung:
betreuer = rechnung.objekt.betreuer_vertretung
rechnung.zugewiesen_an = betreuer
else:  # 2b oder 3
rechnung.routing_ziel = 'frontoffice'
rechnung.zugewiesen_an = None  # Queue
return
def konfidenz_min(rechnung) -> float:
return min(
rechnung.erkennungs_konfidenz.get('kreditor', 0),
rechnung.erkennungs_konfidenz.get('objekt', 0),
rechnung.erkennungs_konfidenz.get('konto', 0),
)
Konfidenz-Schwelle nur für Auto-Buchung
Stufe 1 mit Konfidenz < 95% überspringt nicht das Auto-Buchen — sie geht in den ersten manuellen Limit-Schritt (typischerweise Sachbearbeiter), auch wenn der Betrag < Auto-Limit ist. Damit wird verhindert, dass eine 'gerade so erkannte' Rechnung ohne Sichtung gebucht wird.

def darf_direkt_freigeben(rechnung: Rechnung, bearbeiter: User) -> bool:
grenzen = rechnung.objekt.zahlungsfreigabe_grenzen
stufe = ermittle_freigabestufe(rechnung.betrag, grenzen)
if stufe['rolle'] == 'auto':
# Direktes Buchen — nur wenn Konfidenz nun ≥ 95% (durch Identifikation
# erreicht: Konto-Auswahl gibt Konfidenz 1.0, Kreditor/Objekt durch
# Bearbeiter-Bestätigung 1.0)
return True
if stufe['rolle'] == 'sachbearbeiter':
# Frontoffice-User dürfen wie Sachbearbeiter freigeben.
# Objektbetreuer nur wenn er auch Sachbearbeiter für das Objekt ist.
if bearbeiter.has_role('Frontoffice'):
return True
return bearbeiter in rechnung.objekt.sachbearbeiter.all()
if stufe['rolle'] == 'geschaeftsfuehrer':
return bearbeiter.has_role('Geschaeftsfuehrer')
return False
7. Lernlogik
7.1 Trigger
Trigger A — Identifikation in Prüffall (egal ob Objektbetreuer oder Frontoffice): erzeugt Regel mit erstellt_aus='pruefung'.
Trigger B — Konto-Korrektur in der Freigabe: alte Regel → veraltet; neue Regel mit erstellt_aus='freigabe_korrektur'.
Trigger C — Manuelle Erfassung in Stufe 3 (neuer Kreditor): Anlage Person mit Rolle Dienstleister; Regel mit erstellt_aus='manuell'.
7.2 Idempotenz und Konflikt
Doppelte Bestätigung gleiche Konto-Wahl: keine neue Regel, trefferzahl++ + letzte_anwendung.
Pro (kreditor, objekt, leistungstext_hash) max. 1 aktive Regel — unique_together-Constraint.
Korrektur ersetzt: alte Regel → veraltet, neue Regel → aktiv.
Opt-out: Checkbox "Einzelfall — keine Regel speichern" am Speichern-Dialog.

Pfad 6 — Stufe 2b (nur Kreditor erkannt, Objekt fehlt) → Routing Frontoffice (zugewiesen_an=NULL); Frontoffice-User lädt Inbox, sieht Rechnung → Lock → identifiziert.

Pfad 9 — Frontoffice-Lock: User A öffnet, User B sieht "in Bearbeitung"; Lock läuft ab → User B kann übernehmen.
Pfad 10 — Lernen aus Korrektur: Konto-Änderung in Freigabe → alte Regel veraltet, neue Regel mit erstellt_aus='freigabe_korrektur'.
Pfad 11 — Idempotenz: zweimalige gleiche Bestätigung → trefferzahl++.
Pfad 12 — Frontoffice darf NICHT in Stufe-2a-Prüffälle anderer Objekte greifen — nur eigene Inbox + Frontoffice-Queue.
10.3 Edge Cases
Konto eines anderen Objekts darf nicht zugewiesen werden — Validation Error.
Auto-Limit konfiguriert, aber Konfidenz exakt 0.95 → zählt als auto-fähig (≥, nicht >).
Stufe 1 ohne Auto-Limit konfiguriert (zahlungsfreigabe_grenzen ohne 'auto'-Stufe) → immer in_pruefung, Konfidenz-Schwelle irrelevant.
Veraltete Regel ignoriert — aktive Regel gewinnt.
Kreditor mit mehreren IBANs — erste Treffer-IBAN gewinnt.

12. Wichtige Hinweise fuer die Implementierung
ROUTING NACH ERKANNTER DIMENSION
Stufe 2 ist nicht monolithisch. Das Routing entscheidet anhand der erkannten Dimension: Objekt erkannt (mit oder ohne Kreditor) → Objektbetreuer. Nur Kreditor erkannt → Frontoffice. Stufe 3 → Frontoffice.
95% NUR FÜR AUTO-BUCHUNG
Die 95%-Schwelle ist ein zusätzliches Kriterium AN DER AUTO-STUFE — nicht an Stufe 1 als Ganzes. Stufe-1-Rechnungen mit Konfidenz < 95% laufen ganz normal in den Limit-Workflow (mit manueller Sichtung), auch wenn der Betrag unter dem Auto-Limit liegt. Begründung: 'gerade so erkannt' soll nicht ohne Sichtung gebucht werden.
FRONTOFFICE IST GETEILTE INBOX
Mehrere User können die Rolle Frontoffice haben. Soft-Lock von 5 Minuten beim Bearbeiten verhindert Doppelbearbeitung. Heartbeat alle 30 Sekunden verlängert den Lock während aktiver Bearbeitung.
NACH IDENTIFIKATION IMMER LIMIT-WORKFLOW
Egal ob Objektbetreuer (2a), Frontoffice (2b/3) oder Doppelfunktion-Freigabe — nach Identifikation greift der reguläre Limit-Workflow gemäss zahlungsfreigabe_grenzen. KEIN Rückweg zum Objektbetreuer aus Frontoffice und kein Sonderpfad.
OBJEKTSPEZIFISCHE REGELN, SOFORTIGES LERNEN
Match-Regeln gelten je (kreditor, objekt, leistungstext_hash). Jede manuelle Korrektur lernt sofort, mit optionaler Opt-out-Checkbox.
