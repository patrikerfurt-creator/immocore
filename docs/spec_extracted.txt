1. Zweck dieses Dokuments
Dieses Dokument enthält den vollständigen Claude Code Implementierungsprompt für das Modul „Rechnungserkennung mit Lernlogik“. Es erweitert den bereits in Phase 3 implementierten Rechnungs-Lifecycle (Ausgangsspezifikation v1.1, Kap. 7) um:
drei klar getrennte Erkennungsstufen mit eindeutiger Routing-Logik
ein lernendes Regelwerk für die automatische Buchungskonto-Zuordnung
ein neues Pflichtfeld Objekt.betreuer als Routing-Ziel für Prüffälle und nicht erkannte Rechnungen
Das Modul ist additiv: Bestehende Status (in_pruefung, freigegeben, gebucht, bezahlt, abgelehnt) und der Mechanismus zahlungsfreigabe_grenzen bleiben unverändert. Ergänzt werden Erkennungs-Felder, Match-Regeln und ein klar definierter Routing-Schritt zwischen Erkennung und Freigabe.
2. Workflow im Überblick
Der gesamte Ablauf von der Rechnungseingangs-Pipeline bis zur Buchung folgt einem klaren Entscheidungsbaum. Erkennung und Freigabe sind zwei sauber getrennte Phasen:
2.1 Phase A — Erkennung (automatisch)
Die Erkennungs-Pipeline prüft jede eingehende Rechnung auf drei Dimensionen und ordnet sie genau einer von drei Stufen zu. Dies ist ein rein technischer Vorgang ohne Nutzerinteraktion.
2.2 Phase B — Limit-basierter Freigabeworkflow (nur Stufe 1)
Eine Rechnung im Status erkannt durchläuft den bereits in Phase 3 implementierten Freigabeworkflow gemäß dem Objekt-spezifischen JSONField zahlungsfreigabe_grenzen. Es gibt keinen zusätzlichen Sichtungsschritt durch den Objektbetreuer:
2.3 Phase C — Objektbetreuer-Routing (Stufe 2 + 3)
Prüffälle und nicht erkannte Rechnungen gehen unabhängig vom Betrag immer zuerst an den im Objekt hinterlegten Objektbetreuer. Er erscheint dort unter "Rechnungen zur Prüfung" im Dashboard. Der Objektbetreuer hat eine Doppelfunktion:
Identifizieren — fehlende Dimensionen ergänzen (Kreditor anlegen/zuordnen, Objekt zuweisen, Buchungskonto wählen).
Entscheiden im eigenen Limit — ist der Rechnungsbetrag innerhalb seiner Freigabekompetenz, kann er die Rechnung in einem Schritt identifizieren UND freigeben.
Über Limit — nach Identifikation läuft die Rechnung in den regulären Limit-Workflow (Eskalation an GF).
Ablehnen — jederzeit mit Begründung möglich, ohne Lerneffekt.
Ändern — OCR-Felder (Betrag, Datum, Leistungstext) können korrigiert werden; bei Konto-Korrektur entsteht eine Match-Regel (Lernen).

7. Lernlogik — Trigger und Regeln
Es gibt drei Trigger, die eine RechnungsMatchRegel erzeugen oder aktualisieren. Alle wirken sofort, ohne weiteres Bestätigungs-UI.
7.1 Trigger A — Identifikation im Prüffall
Objektbetreuer öffnet Rechnung im Status pruefung_match oder nicht_erkannt.
Er ergänzt fehlende Dimensionen — typischerweise das Buchungskonto.
Klick auf "Identifizieren + Speichern" oder "Identifizieren + Freigeben" erzeugt eine RechnungsMatchRegel mit erstellt_aus='pruefung'.
7.2 Trigger B — Korrektur in der Freigabe
Buchhalter oder Freigeber öffnet Rechnung im Status in_pruefung oder freigegeben.
Er ändert das Buchungskonto auf einen abweichenden Wert.
Speichern: alte Regel → status=veraltet; neue Regel mit erstellt_aus='freigabe_korrektur'.
7.3 Trigger C — Manuelle Erfassung (Stufe 3, neuer Kreditor)
Betreuer erfasst Rechnung im Status nicht_erkannt vollständig manuell.
Falls Kreditor neu: Anlage als Person mit Rolle Dienstleister.
Bestätigung erzeugt Regel mit erstellt_aus='manuell'.
7.4 Idempotenz und Konflikt
Doppelte Bestätigung gleiche Konto-Wahl: keine neue Regel — trefferzahl++ und letzte_anwendung aktualisieren.
Pro (kreditor, objekt, leistungstext_hash) maximal 1 aktive Regel — unique_together-Constraint erzwingt es.
Korrektur ersetzt: alte aktive Regel wird auf veraltet gesetzt, neue Regel wird aktiv.
Opt-out: Checkbox "Nicht als Regel speichern (Einzelfall)" am Speichern-Dialog — deaktiviert das Lernen für diese eine Korrektur.

10.2 Integrations-Tests — Workflow-Pfade
Pfad 1 — Stufe 1 unter Limit: erkannte Rechnung 250 € → sofort gebucht, kein Nutzer involviert.
Pfad 2 — Stufe 1 über Limit: erkannte Rechnung 5.000 € → in_pruefung, zugewiesen_an = GF, KEIN Routing zum Objektbetreuer.
Pfad 3 — Stufe 2 in Betreuer-Limit: Prüffall 800 €, Betreuer = Sachbearbeiter → Identifizieren+Freigeben verfügbar → freigegeben → gebucht.
Pfad 4 — Stufe 2 über Betreuer-Limit: Prüffall 12.000 €, Betreuer = Sachbearbeiter → Identifizieren+Freigeben gesperrt → nach Speichern automatisch in_pruefung mit Eskalation an GF.
Pfad 5 — Stufe 3: nicht erkannt → Routing an Objektbetreuer → manuelle Erfassung → Regel mit erstellt_aus='manuell'.
Pfad 6 — Lernen aus Korrektur: Stufe 1 mit Konto-Korrektur in Freigabe → alte Regel veraltet, neue Regel mit erstellt_aus='freigabe_korrektur'.
Pfad 7 — Idempotenz: zweimalige gleiche Bestätigung → trefferzahl++, keine zweite Regel.
Pfad 8 — Vertretung: Betreuer.abwesend = True → Routing an betreuer_vertretung.
Pfad 9 — Stufe 3 ohne Objekt: Routing an mandant_default_pruefer (keine Objekt-Betreuer-Auflösung möglich).
Pfad 10 — Ablehnung: KEIN Lerneffekt, kein Regel-Eintrag.
10.3 Edge Cases
Konto eines anderen Objekts darf nicht zugewiesen werden — Validation Error.
Veraltete Regel wird bei Erkennung ignoriert; aktive Regel gewinnt.
Mehrere Objekte gleicher Anschrift (Bauabschnitt A/B) — ohne Verwendungszweck-Match → Stufe 2.
Kreditor mit mehreren IBANs — alle werden geprüft, erste Treffer-IBAN gewinnt.

def normalisiere_leistungstext(text: str) -> str:
    text = text.lower()
    text = re.sub(r'\d{1,2}[./]\d{1,2}[./]\d{2,4}', '', text)  # Datum
    text = re.sub(r'\b(q[1-4]|kw\s?\d{1,2})\b', '', text)     # Quartal/KW
    text = re.sub(r'(rg|re|rechnung|beleg)[-.\s]?\d+', '', text)  # Belegnr
    text = re.sub(r'[^a-z0-9\u00e4\u00f6\u00fc\u00df ]+', ' ', text)
    text = re.sub(r'\s+', ' ', text).strip()
    text = ' '.join(t for t in text.split() if t not in DE_STOPWORDS)
    return text
 
def leistungstext_hash(text: str) -> str:
    return hashlib.sha256(
        normalisiere_leistungstext(text).encode('utf-8')
    ).hexdigest()

def fuehre_erkennung_aus(rechnung: Rechnung) -> Rechnung:
    log = RechnungsErkennungsLog(rechnung=rechnung)
 
    kreditor_match = match_kreditor(rechnung)
    objekt_match   = match_objekt(rechnung)
 
    # Konto nur sinnvoll wenn Kreditor & Objekt eindeutig
    konto_match, regel_treffer = MatchResult.empty(), None
    if kreditor_match.eindeutig and objekt_match.eindeutig:
        text_hash = leistungstext_hash(rechnung.leistungstext or '')
        regel_treffer = RechnungsMatchRegel.objects.filter(
            kreditor=kreditor_match.kandidat,
            objekt=objekt_match.kandidat,
            leistungstext_hash=text_hash,
            status='aktiv',
        ).first()
        if regel_treffer:
            konto_match = MatchResult.eindeutig(regel_treffer.buchungskonto, 1.0)
        else:
            konto_match = match_konto_historie(
                kreditor_match.kandidat, objekt_match.kandidat
            )  # Konfidenz <= 0.7 — Vorschlag, nicht eindeutig
 
    # Stufenableitung
    if (kreditor_match.eindeutig and objekt_match.eindeutig
            and konto_match.eindeutig):
        rechnung.status = 'erkannt'
        rechnung.erkennungs_stufe = 1
    elif kreditor_match.eindeutig or objekt_match.eindeutig:
        rechnung.status = 'pruefung_match'
        rechnung.erkennungs_stufe = 2
    else:
        rechnung.status = 'nicht_erkannt'
        rechnung.erkennungs_stufe = 3
 
    rechnung.kreditor      = kreditor_match.kandidat if kreditor_match.eindeutig else None
    rechnung.objekt        = objekt_match.kandidat   if objekt_match.eindeutig   else None
    rechnung.buchungskonto = konto_match.kandidat    if konto_match.eindeutig    else None
    rechnung.match_regel   = regel_treffer
    rechnung.erkennungs_konfidenz = {
        'kreditor': kreditor_match.konfidenz,
        'objekt':   objekt_match.konfidenz,
        'konto':    konto_match.konfidenz,
    }
 
    # Routing (Phase B oder C)
    route_rechnung(rechnung)
 
    rechnung.save(); log.save()
    return rechnung

def route_rechnung(rechnung: Rechnung) -> None:
    if rechnung.status == 'erkannt':
        # Phase B — Limit-basierter Freigabeworkflow
        grenzen = rechnung.objekt.zahlungsfreigabe_grenzen
        stufe = ermittle_freigabestufe(rechnung.betrag, grenzen)
 
        if stufe['rolle'] == 'auto':
            buche_rechnung(rechnung)  # Status -> 'gebucht'
            return
 
        rechnung.status = 'in_pruefung'
        rechnung.zugewiesen_an = ermittle_freigabeperson(
            rechnung.objekt, stufe
        )
        starte_eskalations_task(rechnung, stufe['frist_tage'])
        return
 
    # Phase C — Prüffall oder nicht erkannt
    if rechnung.status in ('pruefung_match', 'nicht_erkannt'):
        betreuer = rechnung.objekt.betreuer if rechnung.objekt else None
        # Bei nicht erkannt ohne Objekt: Fallback auf
        # Default-Inbox (Mandant-Konfiguration)
        if betreuer is None:
            betreuer = mandant_default_pruefer()
        if betreuer.abwesend and rechnung.objekt:
            betreuer = (rechnung.objekt.betreuer_vertretung
                        or betreuer)
        rechnung.zugewiesen_an = betreuer
        # KEIN Eskalations-Task in Phase C — erst nach
        # Identifikation gilt der Limit-Workflow.
        return

def darf_betreuer_direkt_freigeben(
    rechnung: Rechnung,
    betreuer: User,
) -> bool:
    grenzen = rechnung.objekt.zahlungsfreigabe_grenzen
    stufe = ermittle_freigabestufe(rechnung.betrag, grenzen)
 
    # Auto-Stufe: nichts freizugeben
    if stufe['rolle'] == 'auto':
        return True   # Button-Label wird zu 'Identifizieren + Buchen'
 
    # Sachbearbeiter-Stufe: Betreuer muss als Sachbearbeiter
    # für dieses Objekt konfiguriert sein
    if stufe['rolle'] == 'sachbearbeiter':
        return betreuer in rechnung.objekt.sachbearbeiter.all()
 
    # GF-Stufen: Betreuer braucht GF-Rolle
    if stufe['rolle'] in ('geschaeftsfuehrer',):
        return betreuer.has_role('Geschaeftsfuehrer')
 
    return False
